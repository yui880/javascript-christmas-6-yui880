### 함수(메서드) 라인에 대한 기준
- eslint의 rule을 통해 `max-lines-per-function`를 15로 설정해서 15라인이 넘어가지 않도록 신경써주었다.
- 15 라인이 넘지 않게 작성하라는 기준은 내가 한 메서드에 너무 많은 작업을 하지 않는지 돌아보게 해주는 이정표가 되어준다고 생각한다.
- 컨트롤러에서 View의 함수를 호출하기만 하는 메서드가 있었는데 너무 메서드가 길어져서 어떻게 해야할지 고민이 많았었다. 해당 메서드의 역할이 출력이라고 생각했기 때문에 하나의 역할만 한다고 생각했기 때문이다. 그러나 15라인 오류를 확인하고서 정말 나눌 방법이 없을까 돌아보는 계기가 되어주었다.
  - 해당 로직의 가독성적인 측면이나 프로그램의 흐름에 따라 이벤트 적용 전후로 나눠주는 것이 좋을 것이라고 생각해서 메서드를 분리하였고, 더 나은 메서드를 작성할 수 있었다


### 발생할 수 있는 예외 상황에 대해 고민한다.
- 어떤 예외 상황을 찾으면, 해당 예외 상황을 고려하기 위해 필요한 다른 기능을 확인하는 방식으로 더 자세한 예외 사항들을 확인해나갔다.
- 해당 숫자가 1-31 사이의 숫자가 아니면 예외 처리 <- 해당 값이 정수인지 확인 <- 해당 값이 숫자인지 확인 
- 예외 상황에 대한 테스트 코드를 꼼꼼하게 짜면서 놓친 예외 상황이 있을지 확인하였다.


### 비지니스 로직과 UI 로직을 분리한다.
- 이번 요구사항에는 InputView와 OutputView 객체가 주어졌기 때문에 로직을 view와 domain로 나누기 수월했다.
- mvc 패턴을 적용하여 과제를 수행하였는데, view는 입출력에 관련된 일만 수행하도록 하고 domain에서 비지니스 로직이 구현될 수 있도록 하였다.
- 이렇게 작성하니 각각의 모델이나 뷰가 수정되어도 서로에게 큰 영향을 미치지 않아서 코드를 리팩토링하는데도 편리하였다.
- 특히 view에서 값을 받아서 해당 값을 정제해 출력할 수 있도록 하였다.
  - 출력될 값의 형식을 model(domain)에서 정해버리게 되면 핵심 로직이 수정될 때 출력될 형식을 view가 아니라 domain에서 수정되는 상황이 발생한다.
  - 각자의 로직을 확실히 분리하기 위해선 이렇게 구현하는 것이 옳다고 생각하였다. 


### 객체의 상태 접근을 제한한다.
- 필드는 private(#)을 사용해서 외부에서 접근할 수 없게 해주었다.
- 필드를 getter로 외부에 제공하는 것도 지양하였다.
  - 해당 값이 외부로 나가게 되면, 해당 값이 조작되어서 또 다르게 사용될 가능성이 있기 때문이다. 해당 상태는 해당 객체 안에서만 다뤄질 수 있게 하였다.
  - 완전히 사용하지 않은 것은 아니다. 해당 값을 출력해야하는 경우에는 getter를 사용하였다.
  - 해당 값을 처리해서 특정 조건에 만족하는지/아닌지, 또는 해당 값을 조작해서 나온 최종 결과를 제공할 수 있게 노력하였다. -> getter 대신 메세지 보내기 


### 객체는 객체스럽게 사용한다.
- 상태는 객체 내부에서 관리하고 조작되어야한다.
- 외부에서 조작되는 것을 막기 위해서 getter를 지양하고, 메세지를 주고 받을 수 있도록 만들어주었다.


### 필드의 수를 줄이기 위해 노력한다.
- 해당 객체가 관리해야하는 핵심 데이터는 필드에 저장하고, 해당 데이터를 조작해서 만들 수 있는 값은 필드에 저장하지 않았다.
  - 메서드를 통해 해당 값을 얻을 수 있게 하였다.
  - 보통 특정 조작을 통해 얻은 값은 그때만 사용되고, 해당 객체 내에서 재사용되지 않기 때문이다. 
  - 재사용 되는 핵심 데이터만 필드에 저장


### 성공하는 케이스 뿐만 아니라 예외에 대한 케이스도 테스트한다.
- 모든 케이스에 대해 성공/실패 테스트를 작성하기 위해 노력하였다.
- 실패하는 테스트 코드를 작성하면서 미처 확인하지 못했던 예외나 오류를 확인하는 경우가 많았다.
  - 성공하는 테스트 코드만 작성하면 예외 상황에 제대로 대응하는지 확인하기엔 부족하다고 생각했다.
  - 할 수 있는 한 모든 케이스를 고려하기 위해 노력하였다.


### 테스트 코드도 코드다 
- test.each()를 사용해 하나의 로직을 가지고 다양한 값을 테스트 하도록 작성하였다.
  - 범위를 다루는 메서드의 경우 경계값에 대한 테스트를 진행해야 하는데 이럴 때 같은 로직으로 다양한 값을 확인할 필요가 있었다. 
  - test.each()를 사용하면 로직을 한번만 작성할 수 있어 테스트 코드 작성 유지비를 줄일 수 있었다.

- 코드를 이해하기 쉽도록 given-when-then 방식을 사용하였다.
  - 테스트 코드를 읽다보면 어떤 로직을 수행하고 있는 것인지, 어떤 값이 결과고, 어떤 값이 예상값인지 구분하기 어려운 경우가 있다.
  - 그래서 test.each를 통해 입력, 결과를 한번에 작성하고 해당 값을 받아와서 넣어서 테스트 구현 로직 자체가 짧아지게 하였다.(값은 test.each(값)에 작성하고, 로직은 내부 함수 안에서 작성해서 둘을 분리했다는 의미)
  - 그렇게 한 후 given-when-then으로 코드를 분리함으로서 한눈에 테스트 코드가 이해될 수 있게하였다. 


### 테스트를 위한 코드는 구현 코드에서 분리되어야 한다
- private 메서드에 대한 테스트는 작성하지 않았다.
  - 테스트를 위해서 private한 메서드를 public으로 바꾼다는 것은 옳지 못하다고 생각한다.
  - 그래서 직접 그 메서드를 테스트하지 않고, 다른 메서드들 통해 간접적으로 테스트 될 수 있게 하였다.


### 단위 테스트하기 어려운 코드를 단위 테스트하기
- 단위 테스트하기 어려운 코드를 보면서 더 나은 방향으로 코드를 리팩토링 할 수 있었다.
  - mock : 테스트를 작성하면서 특정 메서드나 객체를 mock으로 대체해서 사용할 수 있다.
    - Promotion 테스트를 작성하면서 해당 클래스에서 사용되는 Discount와 Gift 객체를 mock해서 테스트하려고 했다.
    - 하지만 각 객체를 Promotion 내부에서 생성하기 때문에 mock으로 대체하기 어려웠다. -> 문제 파악 -> 의존성 주입을 사용한 리팩토링 진행
  - 의존성 주입: 의존성을 외부에서 주입받아서 사용하는 방법
      - 의존성 주입이라는 것을 학습하고 클래스에 적용해보았는데, constructor의 인자로 다른 클래스 객체를 전달해주니까 훨씬 테스트를 수행하기 수월해졌다.

## 🚀 목표
### ⭐️ 클래스(객체)를 분리하기
클래스를 분리하는데 기준을 세우고, 해당 기준에 맞게 분리할 수 있도록 노력하였다.
1. **단일 책임 원칙** : 클래스의 역할을 명확히 하고, 해당 역할 이외의 작업을 하고 있다면 클래스를 분리해서 하나의 역할(책임)만 수행하도록 하였다.
   - `단일 책임` : 클래스가 변경되어야 하는 이유가 하나여야 함 -> Discount 클래스 : 할인 방법에 대한 클래스 -> 할인 로직을 수정할 때 Discount만 수정하면 됨 
   - 단, 클래스가 너무 많이 분리되지 않게 하기 위해 노력하였다. 책임은 보는 관점에서 다양해질 수 있는데, 너무 작은 책임으로 나누면 오히려 코드가 복잡해져 추후 수정이 불편할 것이라고 생각했다.
   - 클래스를 보았을 때 무슨 역할을 하는지 바로 확인할 수 있도록 클래스 명을 명확하게 짓기 위해 노력하였고, 다루는 필드를 1,2개로 설정해서 로직이 복잡하지 않도록 관리하였다.


2. **상태는 해당 객체 안에서 다루게 하기** : 객체 안에서 다루는 상태는 객체가 스스로 다루고 관리할 수 있어야 한다.
   - 상태 값을 다른 객체(클래스)에서 다루게 된다면, 해당 값을 확인하는데서 끝나지 않고 해당 값이 다른 객체에서 조작될 가능성이 높다. 
   - 이번 과제에서도 날짜에 대한 값을 날짜와 관련이 없는 Discount 클래스에서 계속 조작하는 상황이 발견되어서, 날짜에 대한 값을 Day 클래스로 따로 분리하고 해당 객체에 메세지를 보내 내가 원하는 날짜인지만 확인 할 수 있게 하였다.
   - 객체는 스스로의 상태를 스스로가 관리할 수 있어야 한다. 그렇게 될 수 있도록 클래스를 분리하기 위해 노력해보았다.

### ⭐️ 작고 유용한 프로그램에서 점점 큰 프로그램으로 발전시켜나가기 (가장 핵심 기능부터 만들기)
- 코수타에서 코치님의 의견을 듣고, 이번 과제에서는 프로그램의 순서대로 개발하는 것이 아니라 가장 핵심적인 부분부터 만들기 시작하였다.
- 핵심 기능이 동작하는 프로그램을 만들고 -> 해당 프로그램에 다른의 기능을 추가해가며 확장해 나갔다.
- 이벤트 플래너 프로그램의의 핵심은 `할인`이라고 생각하였다. 
  - 그래서 특정 값이 주어졌을 때 그 값을 할인한 결과를 도출하는 프로그램부터 제작하기 시작했다.(Discount)
  - 그 다음 해당 프로그램에서 적용될 수 있는 할인 이밴트들을 구현하였고, 그 다음엔 할인 이벤트와 유사한 증정 이벤트를 구현하였다.
  - 이런 이벤트들을 모아 관리하는 기능을 만들고 -> 주문한 메뉴에 대해 이벤트를 적용할 수 있도록 하고 -> 이 결과를 도출 할 수 있도록하며 점차 발전시켜나갔다.
- 그래서 모든 단계에서 해당 프로그램만 동작시켰을 경우에도 원하는 동작을 수행할 수 있도록 그 자체로 유용한 프로그램이 될 수 있게 노력하였다.

## 📍 프로그램 흐름 
1. 식당 방문 날짜 입력받기 (잘못된 입력인 경우 재입력)
2. 주문할 메뉴와 개수 입력받기 (잘못된 입력인 경우 재입력)
3. 주문 메뉴 출력하기
4. 주문 메뉴에 따른 총주문 금액 출력하기 
5. 주문 금액과 메뉴, 날짜에 따라서 적용되는 할인 이벤트 확인하기 
6. 적용되는 증정 이벤트에 따라서 증정품 구하기
7. 적용되는 할인 이벤트에 따라서 각 할인 금액 구하기
8. 적용되는 할인,증정 이벤트의 이름과 할인 금액 출력하기 
9. 총 혜택 금액 계산하고 출력하기 (할인 금액 + 증정품 가격)
10. 할인 후 예상 결제 금액 계산하고 출력하기 (총 금액 - 할인 금액)
11. 총 혜택 금액에 따른 뱃지 출력하기 
12. 프로그램 종료

- [x] 할인 이벤트 결과와 할인 금액 출력하기

```
┗ src
  ┣ constants
  ┃  ┗ constant.js         // 프로그래밍 중 사용되는 숫자를 상수로 정리
  ┃  ┗ event.js            // 이벤트와 관련된 상수 정리
  ┃  ┗ menu.js             // 메뉴와 관련된 상수 정리(메뉴 이름, 카테고리, 가격)
  ┃  ┗ message.js          // 출력시 필요한 메세지를 상수 정리 
  ┣ controller
  ┃  ┗ EventPlanner.js     // 이벤트 플래너 실행을 책임지는 컨트롤러 클래스
  ┣ domain
  ┃  ┣ Day.js              // 날짜를 저장하고 관리하는 클래스
  ┃  ┗ Discount.js         // 할인 이벤트를 관리하는 클래스
  ┃  ┗ Gift.js             // 증정 이벤트를 관리하는 클래스
  ┃  ┗ Product.js          // 주문한 메뉴를 관리하는 클래스
  ┃  ┗ Promotion.js        // 이벤트 전체(할인 이벤트 + 증정 이벤트)를 관리하는 클래스
  ┣ view
  ┃  ┣ InputView.js        // 입력에 사용되는 메서드들의 모음
  ┃  ┗ OutputView.js       // 출력에 사용되는 메서드들의 모음
  ┣ validator
  ┃  ┣ Validator.js        // 검증에 사용되는 메서드들의 모음
  ┃  ┗ ValidationError.js  // [ERROR] 키워드를 위한 커스텀 에러 클래스    
  ┣ App.js                 // 프로그램 실행 역할 
  ┗ index.js  
```
